# 合并与冲突策略

> 仿 Trace2Skill 的无冲突层次化合并，扩展至 4 维（procedure + interaction + memory + personality）。
> 补充参考：掘金 Skill 工程指南的「信息保质期」。

## 1. 证据分级（从高到低）

| 标签 | 含义 | 来源示例 |
|------|------|---------|
| `verbatim` | 可指回具体消息/文件的摘抄或原话 | 聊天记录、邮件原文、社交媒体帖子 |
| `artifact` | 文档/作品/公开内容中的客观陈述 | 技术方案、博客文章、演讲文字版 |
| `impression` | 数据提供者补充的主观印象 | 「我觉得 TA...」 |

合并规则：**verbatim > artifact > impression**。

## 2. 同级矛盾的处理

当两条同级别证据互相矛盾时：
- 若可判断时间先后 → **较新者为主**，旧版标注已过期
- 若无时间线索但场景不同 → **并列保留**，标明适用场景
- 若无法裁决 → 进入「待决冲突表」

## 3. 跨维度一致性检查

合并时需检查 4 个维度之间的一致性：
- `procedure` 中的决策模式 vs `personality` 中的价值观：是否吻合？
- `interaction` 中的沟通风格 vs `memory` 中的自述：是否矛盾？
- 发现跨维度矛盾时，同样写入 `conflicts.md`

## 4. 待决冲突表（`conflicts.md`）

```markdown
# 待决冲突

## C1：<冲突主题>
- 说法 A：...（来源、日期、证据级别、所属维度）
- 说法 B：...（来源、日期、证据级别、所属维度）
- 临时裁定：<采用 A / 采用 B / 并列 / 待用户决定>
- 裁定理由：<时间较新 / 场景更匹配 / 用户指定>
```

默认：**较新且场景明确者优先**。

## 5. 增量合并（追加材料时）

1. 读取现有各维度文件。
2. 对新材料执行 intake-protocol 的元数据标注。
3. 按维度逐条比对：**已有则补充或更新**；**新维度则追加**。
4. 矛盾项按冲突规则处理。
5. 更新 `manifest.json`。

## 6. 按角色的维度组合

| 角色 | procedure | interaction | memory | personality |
|------|-----------|-------------|--------|-------------|
| 自己 | ✅ | ✅ | ✅ | ✅ |
| 同事 | ✅（核心） | ✅（核心） | ❌ | 可选 |
| 导师 | 可选 | ✅ | 可选 | ✅ |
| 亲人 | 可选 | ✅ | ✅（核心） | ✅ |
| 伴侣 | ❌ | ✅（核心） | ✅（核心） | ✅ |
| 朋友 | 可选 | ✅（核心） | ✅ | ✅ |
| 公众人物 | 可选 | ✅ | 可选 | ✅ |

## 7. 与最终 SKILL.md 的关系

`SKILL.md` 正文应极短（< 100 行），只包含 frontmatter + 运行规则 + 局限声明。
长内容在独立文件中，按需加载。
